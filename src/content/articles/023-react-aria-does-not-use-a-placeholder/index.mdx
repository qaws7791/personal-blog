React Aria의 TextField 컴포넌트는 placeholder 속성을 지원하지 않습니다. 이는 접근성(Accessibility) 문제를 고려한 결정으로 보입니다. placeholder는 사용자가 입력을 시작하면 사라지기 때문에, 시각 장애인이나 인지 장애를 가진 사용자에게 입력 필드의 목적을 지속적으로 전달하지 못할 수 있습니다. 예를 들어, 스크린 리더(screen reader)에서 placeholder가 제대로 읽히지 않거나, 사용자가 필드의 의도를 잊어버릴 위험이 있습니다. 대신 React Aria는 명확한 `<Label>` 요소와 `<Text slot="description">` 같은 추가 설명을 통해 필드의 목적을 항상 보이게 하거나 접근 가능하게 유지하도록 설계되었습니다. 이는 WCAG(Web Content Accessibility Guidelines) 지침과 일치하며, 모든 사용자에게 더 포괄적인 경험을 제공하기 위한 접근입니다.

사용자 관점에서 이 결정을 분석해 보자면, 전체적으로 잘한 결정이라고 평가할 수 있습니다. 아래에서 장단점을 나누어 설명하겠습니다.

### 장점 (긍정적 측면)

- **접근성 향상**: placeholder에만 의존하면, 입력 후 힌트가 사라져 혼란을 줄 수 있습니다. 특히 시각 장애인(스크린 리더 사용자)이나 노인, 인지 부하가 높은 사용자에게는 label이 항상 존재하는 것이 더 안전하고 직관적입니다. 이는 모든 사용자가 앱을 더 쉽게 사용할 수 있게 하여 포괄성을 높입니다.
- **지속적인 가이드 제공**: placeholder는 일시적ですが, label은 영구적입니다. 사용자가 입력 중에 필드의 목적을 다시 확인할 수 있어 실수를 줄이고 사용자 경험(UX)을 개선합니다. 예를 들어, 복잡한 폼에서 여러 필드를 채울 때 유용합니다.
- **법적/표준 준수**: 많은 앱이 접근성 법규(예: ADA in US)를 준수해야 하며, placeholder만으로는 label을 대체할 수 없다는 WCAG 기준을 따르는 것이 장기적으로 개발자와 사용자 모두에게 이득입니다.

### 단점 (부정적 측면)

- **익숙함의 상실**: 많은 사용자(특히 일반 웹 사용자)는 placeholder를 통해 빠르게 힌트를 확인하는 데 익숙합니다. 이는 미니멀한 디자인에서 공간을 절약하고 시각적으로 깔끔하게 보이게 합니다. placeholder가 없으면 필드가 더 "텅 빈" 느낌이 들 수 있어, 초기 적응이 필요할 수 있습니다.
- **디자인 유연성 제한**: 모바일이나 작은 화면에서 placeholder는 레이블을 보완하는 역할을 하곤 합니다. 이를 제거하면 디자이너가 추가적인 스타일링(예: floating label)을 적용해야 하며, 일부 사용자에게는 과도하게 복잡해 보일 수 있습니다.
- **특정 시나리오에서의 불편**: 검색 바나 간단한 입력 필드처럼 빠른 상호작용이 필요한 경우, placeholder가 없으면 사용자가 더 많은 텍스트를 읽어야 할 수 있습니다.

결론적으로, React Aria 팀의 결정은 접근성을 최우선으로 한 라이브러리의 철학과 맞물려 잘한 선택입니다. 사용자 관점에서 보면, 단기적으로는 기존 패턴에 익숙한 사람들에게 약간의 불편을 줄 수 있지만, 장기적으로는 더 포괄적이고 안정적인 UX를 제공합니다. 만약 placeholder가 꼭 필요하다면, 개발자가 커스텀으로 추가할 수 있지만, 팀의 의도는 더 나은 표준을 장려하는 데 있습니다.
